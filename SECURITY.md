# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 1.x     | :white_check_mark: |

## Security Model

FITEBOX is designed to operate on a local network (conference venue WiFi or ad-hoc network). It is **not designed for exposure to the public internet**. Key security considerations:

- **Web dashboard** uses a shared key (displayed on the OLED screen) for authentication via HMAC headers.
- **No user accounts or passwords** are stored on the device.
- **Media endpoints** (download, stream) accept key-based query parameter auth for browser compatibility.
- **Docker container** runs in privileged mode for hardware access; recording and streaming share the same container with access only to the volumes they need.

## Reporting a Vulnerability

We take the security of FITEBOX seriously, even though it typically runs on isolated networks.

If you discover a security vulnerability, please **DO NOT** open a public issue.

Instead, please send an email to **juanmi@juanmitaboada.com** with:

- A description of the vulnerability
- Steps to reproduce (if applicable)
- Potential impact assessment
- Suggested fix (if you have one)

We will try to respond within **48 hours** and aim to release a fix within **7 days** for critical issues.

We appreciate responsible disclosure and will credit reporters in the release notes (unless you prefer to remain anonymous).

## Known Limitations

- The shared key is 6 hex characters - sufficient for physical-proximity auth at a conference, but not for internet-facing deployments.
- RTMP stream keys are stored in plaintext on the filesystem (`/tmp/fitebox_stream.json`). This is acceptable for a single-purpose appliance but should be noted.
- The web server uses **self-signed TLS certificates** generated by `bin/setup.sh`. Browsers will show a certificate warning on first connection. For a trusted certificate, replace the generated files or place a reverse proxy with a proper certificate in front of it.
